# Implementation Plan: Apple Health Running Sync

**Branch**: `002-apple-health-sync` | **Date**: 2026-05-03 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `specs/002-apple-health-sync/spec.md`

## Summary

Automatically sync running workout distances from Apple Health into the tracker as individual cumulative log entries, using the HKWorkout UUID as the log entry ID for free deduplication. Sync fires on each app foreground event via the React Native `AppState` API. Offline entries are routed through the existing `insertLogEntry` queue — no new offline infrastructure required.

## Technical Context

**Language/Version**: TypeScript, React Native 0.74.5, Expo SDK 51
**Primary Dependencies**: react-native-health (HealthKit), AppState (React Native built-in), react-native-mmkv (checkpoint storage), existing insertLogEntry + createMetric services
**Storage**: MMKV — sync checkpoint (`'health:last_sync_at'`), cached metric ID (`'health:running_metric_id'`)
**Testing**: Jest + mocked `react-native-health` module
**Target Platform**: iOS 13.4+ (Apple Health is iOS-only; native build required)
**Project Type**: Mobile app (React Native / Expo, prebuild workflow)
**Performance Goals**: Sync completes within 5 seconds of foreground (SC-001); handles up to 90 days of backlog on first sync
**Constraints**: Native build only (no Expo Go); HealthKit entitlement required; must not disrupt other features if permission denied (FR-010)
**Scale/Scope**: Single user per device, personal use

## Phase 0: Research Summary

See [research.md](research.md) for full rationale. Key decisions:

| Unknown | Decision |
|---------|----------|
| HealthKit library | `react-native-health` — battle-tested, supports workout queries with unit selection |
| Deduplication | HKWorkout UUID as log entry `id` — upsert `ignoreDuplicates` handles re-runs for free |
| Foreground trigger | `AppState.addEventListener('change')` — no new dependency |
| Sync checkpoint | MMKV key `'health:last_sync_at'` (ISO string); 90-day fallback on first sync |
| Unit conversion | Request in `'km'` directly; filter < 0.001 km as zero-distance |
| Metric bootstrap | Auto-create `'Running Distance'` (cumulative, weekly) if absent; cache ID in MMKV |

## Phase 1: Design

### `src/services/healthSync.ts`

**`requestHealthPermission(): Promise<boolean>`**
- Calls `AppleHealthKit.initHealthKit` with `Permissions.Workout` read permission
- Returns `true` if granted, `false` if denied or unavailable
- Must not throw — all errors caught and logged

**`syncRunningWorkouts(): Promise<void>`**
- Guard: if `isLocalMode` → return immediately
- Request permission (if not yet granted) → if denied, return silently
- Read `'health:last_sync_at'` from MMKV; default to 90 days ago if null
- Resolve running metric ID: read `'health:running_metric_id'` from MMKV, or query `fetchUserMetrics()` for one named `'Running Distance'`, or create via `createMetric('Running Distance', 'cumulative', 'weekly')`; persist ID to MMKV
- Query HealthKit: `getWorkouts({ type: 'Running', startDate, endDate: now, ascending: true, unit: 'km' })`
- Filter: `distanceKm >= 0.001`
- For each workout (sorted ASC by endDate): call `insertLogEntry({ id: workout.id, metricId, value: distanceKm, loggedAt: endDate, sessionStartAt: startDate })`
- After all entries processed: update `'health:last_sync_at'` to the endDate of the final workout

### `src/hooks/useHealthSync.ts`

- On mount: call `syncRunningWorkouts()`
- Subscribe to `AppState` changes: when state transitions to `'active'`, call `syncRunningWorkouts()`
- On unmount: remove `AppState` listener
- No return value / no UI state — purely a side-effect hook

### `app.json` additions

```json
"ios": {
  "entitlements": {
    "com.apple.developer.healthkit": true
  },
  "infoPlist": {
    "NSHealthShareUsageDescription": "Tracker reads your running workouts to log your distance automatically."
  }

```

### `App.tsx` change

Add `useHealthSync()` call inside the `App` component body (after `sessionReady` guard), so sync fires each time the app foregrounds.
